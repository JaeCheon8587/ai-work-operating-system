# PRD: AI Work Operating System — Phase 1 MVP

## 1. 문서 개요

### 1.1 제품명

**AI Work Operating System**  
가칭: **SprintOps AI**

### 1.2 Phase 1 명칭

**PM Planning & Sprint Assignment**

### 1.3 한 줄 설명

PM이 고객 요구사항을 기반으로 AI와 대화하며 고객 비즈니스 개념, 시스템 도메인 개념, 스프린트 범위, 앱/서비스별 역할을 작성·비판·보완·정리하고, 담당자를 배정한 뒤 저장/확정 시 Slack으로 각 담당자에게 배정 알림과 확인 링크를 발송하며, 관련 PM 설계 문서를 CI를 통해 저장소에 반영하는 시스템.

---

## 2. 제품 배경

현재 팀의 업무는 요구사항을 정리한 뒤 곧바로 개발자에게 세부 작업을 던지는 방식이 아니라, PM이 먼저 시스템 관점에서 요구사항을 해석하고 스프린트 단위로 구조화해야 한다.

PM이 해야 하는 일은 기술 설계가 아니다.

또한 Phase 1의 핵심은 PM이 혼자 문서를 작성하는 것이 아니라, **AI와 함께 작성하고, AI에게 비판받고, AI를 통해 정리하고, AI를 통해 지정된 포맷의 문서로 변환하는 것**이다.

PM은 ChatGPT처럼 자유롭게 내용을 입력할 수 있어야 하며, 시스템은 입력 내용을 구조화된 PM 설계 문서로 변환해야 한다.  
또한 PM은 버튼을 통해 AI에게 현재 내용을 비판하게 하거나, 누락된 개념을 찾게 하거나, 스프린트 범위가 과도한지 검토하게 할 수 있어야 한다.

PM은 다음을 정리해야 한다.

```text
- 고객은 무엇을 원하는가?
- 고객의 비즈니스 안에서는 어떤 개념이 중요한가?
- 이 시스템은 어떤 문제를 해결해야 하는가?
- 이 시스템에는 어떤 도메인 개념이 존재해야 하는가?
- 이번 스프린트에서 어떤 기능을 다룰 것인가?
- 이번 스프린트에서 제외할 것은 무엇인가?
- 어떤 앱/서비스가 관련되는가?
- 각 앱/서비스는 어떤 역할을 맡는가?
- 각 앱/서비스 담당자는 누구인가?
- 담당자에게 전달할 간략한 설명은 무엇인가?
```

이후 PM이 저장 또는 확정 버튼을 누르면 각 담당자에게 Slack 알림이 발송되고, 담당자는 링크를 통해 자신에게 배정된 내용을 확인한다. 동시에 PM 설계 문서는 Markdown 형태로 생성되어 CI를 통해 지정된 저장소에 반영된다.

## 3. Phase 구조

전체 제품은 3개의 개발 Phase로 나눈다.

```text
Phase 1: PM Planning & Sprint Assignment
Phase 2: Worker / Developer Execution
Phase 3: Work Orchestration & Synchronization
```

### 3.1 Phase 1

PM을 위한 단계다.

PM이 요구사항을 정리하고, 고객 비즈니스 개념과 시스템 도메인 개념을 정리하고, 스프린트를 만들고, 앱/서비스별 담당자를 배정한 뒤 Slack으로 알림을 보낸다. 그리고 CI를 통해 레파지토리에 문서들이 반영된다.

### 3.2 Phase 2(추후 설계 예정)

개발자 또는 작업자를 위한 단계다.

담당자는 Slack 링크를 통해 자신에게 배정된 내용을 확인하고, 세부 작업을 수행한다. 이 단계부터 작업 지시서, 세부 Task, 진행 상태, 완료 보고, 블로커 등의 개념이 들어온다.

### 3.3 Phase 3(추후 설계 예정)

PM 설계와 개발자 실행을 연결하는 단계다.

앱/서비스 간 의존성, 병목, 진행 상태, 우선순위 재조정, PM 개입 필요 항목을 AI가 분석하고 보여준다.

## 4. Phase 1 범위

Phase 1은 PM 중심 MVP다.

Phase 1의 핵심은 다음이다.

```text
요구사항 입력 / 자유 작성
↓
AI와 대화하며 요구사항 정리
↓
AI가 PM 설계 초안 생성
↓
PM이 초안 수정
↓
AI가 비판 / 누락 검토 / 대안 제안
↓
PM이 고객 비즈니스 개념 정리
↓
AI가 시스템 도메인 개념 후보 생성
↓
PM이 문제 정의와 해결 방향 확정
↓
AI가 스프린트 범위 / 제외 범위 후보 생성
↓
PM이 스프린트 범위 설정
↓
AI가 참여 앱/서비스와 역할 후보 생성
↓
PM이 앱/서비스별 역할과 담당자 배정
↓
AI가 담당자별 안내 문구와 PM 설계 문서 생성
↓
저장 / 확정
↓
Slack 담당자 알림 발송
↓
담당자 확인 링크 제공
↓
CI를 통한 저장소 반영
```

Phase 1은 작업 지시서 발행 시스템이 아니다.

작업 지시서는 Phase 2 또는 그 아래의 실행 레벨에서 다루는 개념이다.

---

## 5. Phase 1 목표

Phase 1의 목표는 다음이다.

> PM이 요구사항을 기반으로 AI와 함께 고객 비즈니스 개념과 시스템 도메인 개념을 정리하고, 이번 스프린트에서 다룰 기능 범위를 설정한 뒤, 앱/서비스 단위로 역할과 담당자를 배정하고, Slack을 통해 담당자에게 배정 내용을 전달하며, 관련 PM 설계 문서를 CI를 통해 저장소에 반영할 수 있게 한다.

Phase 1의 성공은 PM이 다음 작업을 더 빠르고 일관되게 수행할 수 있는지로 판단한다.

```text
- PM이 자유롭게 입력한 내용을 AI가 PM 설계 포맷으로 정리한다.
- PM이 작성한 내용을 AI가 비판하고 누락 항목을 제안한다.
- 요구사항을 PM 관점에서 정리한다.
- 고객의 비즈니스 개념을 정리한다.
- 시스템 도메인 개념을 정리한다.
- 스프린트 목표와 포함/제외 범위를 정한다.
- 관련 앱/서비스를 식별한다.
- 앱/서비스별 역할을 정리한다.
- 담당자를 배정한다.
- 담당자별 안내 문구를 AI가 생성한다.
- 담당자에게 Slack 알림을 보낸다.
- PM 설계 문서를 Markdown으로 생성한다.
- 생성된 문서를 CI를 통해 저장소에 반영한다.
```

```text
- 서비스 배포 목적의 CI/CD 연동
```

Phase 1은 PM의 설계와 배정까지만 다룬다.

## 8. 핵심 개념

## 8.1 Requirement Input

PM이 시스템에 입력하는 요구사항 또는 요구사항 정리 결과다.

입력 방식은 다음을 지원할 수 있다.

```text
- UI 직접 입력
- 요구사항 정리 스킬 결과 붙여넣기
- API 수신
- Webhook 수신
```

포함 가능한 정보는 다음이다.

```text
- 제목
- 고객명
- 요구사항 본문
- 고객 요청 배경
- 엣지케이스
- 참고 자료
- 우선순위
- 출처
```

## 8.2 AI Planning Copilot

PM이 Phase 1의 모든 작성·정리·검토 작업을 함께 수행하는 AI 협업자다.

AI Planning Copilot은 단순 자동 요약기가 아니다.  
PM이 작성한 내용을 기반으로 다음을 수행해야 한다.

```text
- 자유 입력 내용을 PM 설계 포맷으로 변환
- 고객 요구 요약 생성
- 고객 비즈니스 개념 후보 생성
- 시스템 도메인 개념 후보 생성
- 문제 정의 초안 생성
- 해결 방향 초안 생성
- 스프린트 목표 후보 생성
- 포함 범위 / 제외 범위 후보 생성
- 참여 앱/서비스 후보 생성
- 앱/서비스별 역할 후보 생성
- 담당자별 안내 문구 생성
- 현재 설계 내용 비판
- 누락된 개념 / 리스크 / 예외 케이스 제안
- 과도한 스프린트 범위 경고
- 더 작은 스프린트 범위 제안
- Markdown 문서 생성
```

PM은 AI 결과를 그대로 확정하지 않는다.  
AI는 초안, 비판, 대안, 문서화를 제공하고, 최종 확정은 PM이 수행한다.

## 8.3 AI Interaction

PM과 AI가 협업하는 상호작용 단위다.

지원해야 하는 상호작용은 다음이다.

```text
- Chat 입력
- 특정 섹션에 대한 AI 작성
- 특정 섹션에 대한 AI 비판
- 누락 항목 찾기
- 스프린트 범위 줄이기
- 도메인 개념 후보 생성
- 앱/서비스 역할 후보 생성
- 담당자 안내 문구 생성
- Markdown 문서 변환
```

예시:

```text
PM 입력:
이 요구사항을 고객 비즈니스 개념과 시스템 도메인 개념으로 나눠줘.

AI 결과:
고객 비즈니스 개념, 시스템 도메인 개념, 매핑 후보를 생성한다.
```

```text
PM 버튼 클릭:
현재 스프린트 범위를 비판해줘.

AI 결과:
스프린트 범위가 과도한 부분, 제외해야 할 기능, 다음 스프린트 후보를 제안한다.
```

## 8.4 AI Review Action

PM이 작성한 내용을 AI가 검토하거나 비판하는 기능이다.

AI Review Action은 다음 유형을 지원해야 한다.

```text
- 비판하기
- 누락 찾기
- 범위 줄이기
- 대안 제시
- 도메인 개념 검토
- 스프린트 목표 검토
- 앱/서비스 역할 검토
- 담당자 안내 문구 검토
```

AI Review Action은 PM의 결정을 대체하지 않는다.

## 8.6 Business Concept

고객의 비즈니스 안에서 중요한 개념이다.

예시:

```text
- 데이터 로딩 요청
- 로딩 작업
- 진행 상태
- 실패 상태
- 완료 상태
- 운영자 확인
- 재처리 필요 여부
```

Business Concept은 고객이 이해하는 업무 언어에 가깝다.

---

## 8.7 Domain Concept

고객 비즈니스 개념을 시스템 안에서 관리 가능한 개념으로 정리한 것이다.

예시:

```text
데이터 로딩 요청 → Loading Request
로딩 작업 → Loading Job
진행 상태 → Job Status
실패 원인 → Failure Reason
운영자 확인 → Operator Review
재처리 필요 여부 → Retry Requirement
```

Domain Concept은 API나 DB 설계가 아니다.  
시스템이 어떤 개념을 이해하고 관리해야 하는지를 정리하는 단계다.

## 8.9 App / Service Role

이번 스프린트에서 특정 앱 또는 서비스가 맡는 역할이다.

예시:

```text
Client:
사용자가 데이터 로딩을 요청하고, 로딩 상태를 확인할 수 있는 화면을 제공한다.

API Gateway:
Client 요청을 받아 Loader Master로 전달하고, 상태 조회 응답을 제공한다.

Loader Master:
로딩 작업 생성, 상태 관리, 실행 흐름 제어를 담당한다.

Backend Service:
로딩 작업과 관련된 데이터 저장, 조회, 검증을 담당한다.
```

## 9. PM이 설계하는 것

PM은 기술 설계를 하는 것이 아니다.

Phase 1에서 PM이 설계하는 것은 API, DB, 코드 구조가 아니라 **고객 요구를 시스템의 비즈니스/도메인/스프린트 구조로 정리하는 것**이다.

즉 PM은 “어떻게 구현할 것인가”가 아니라, **무엇을 해결해야 하고, 어떤 개념으로 시스템 안에 자리 잡아야 하며, 이번 스프린트에서 어디까지 다룰 것인가**를 설계한다.

---

### 9.0 모든 PM 작성 작업은 AI와 함께 수행한다

Phase 1에서 PM은 빈 문서를 혼자 채우는 방식으로 일하지 않는다.  
PM은 ChatGPT처럼 자유롭게 입력하고, AI는 이를 각 섹션의 정해진 포맷으로 변환한다.

PM은 각 섹션에서 다음 방식으로 AI와 협업한다.

```text
- 초안 생성: PM이 거칠게 입력하면 AI가 문서 포맷으로 정리한다.
- 비판 요청: PM이 작성한 내용을 AI가 논리적 허점, 누락, 과도한 범위 관점에서 비판한다.
- 누락 탐색: AI가 빠진 비즈니스 개념, 도메인 개념, 예외 케이스, 참여 앱/서비스를 찾는다.
- 대안 생성: AI가 더 작은 스프린트 범위나 다른 앱/서비스 역할 구성을 제안한다.
- 문서 변환: AI가 확정 내용을 Markdown 문서로 변환한다.
- 알림 문구 생성: AI가 담당자별 Slack 안내 문구를 생성한다.
```

예시:

```text
PM 자유 입력:
고객은 데이터 로딩이 어디까지 됐는지 알고 싶어하고, 실패하면 운영자가 확인해야 해.

AI 변환:
- 고객 요구 요약
- 고객 비즈니스 개념
- 시스템 도메인 개념
- 문제 정의
- 스프린트 후보
형태로 정리한다.
```

예시:

```text
PM 버튼 클릭:
이 내용 비판해줘.

AI 검토:
- 스프린트 범위가 과도한 부분
- 빠진 도메인 개념
- 빠진 예외 케이스
- 담당자 배정 전 확인해야 할 앱/서비스
를 제안한다.
```

PM은 AI 제안을 검토하고 최종 반영 여부를 결정한다.

### 9.1 고객은 무엇을 원하는가?

PM은 고객의 요청을 단순 기능명으로 변환하지 않고, 고객이 실제로 얻고 싶은 결과를 정리해야 한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 고객이 명시적으로 요청한 것은 무엇인가?
- 고객이 실제로 해결하고 싶은 문제는 무엇인가?
- 고객이 현재 겪고 있는 불편은 무엇인가?
- 고객은 어떤 업무 상황에서 이 기능을 사용하려 하는가?
- 고객이 성공했다고 느끼는 상태는 무엇인가?
- 고객 요청 중 기능 요구와 업무 요구는 각각 무엇인가?
```

이 단계의 산출물은 다음과 같다.

```text
- 고객 요구 요약
- 고객의 실제 목표
- 현재 문제 상황
- 기대 결과
- 성공 상태 정의
```

예시:

```text
고객 요청:
데이터 로딩 작업의 진행 상태를 확인하고 싶다.

고객의 실제 목표:
데이터 로딩 요청 이후 작업이 정상적으로 진행 중인지, 실패했는지, 완료되었는지를 운영자가 판단할 수 있어야 한다.

현재 문제:
로딩 요청 이후 시스템 내부에서 무슨 일이 일어나는지 보이지 않아, 실패 대응과 고객 응대가 늦어진다.

성공 상태:
운영자는 각 로딩 작업의 현재 상태를 확인하고, 실패 시 원인을 파악할 수 있다.
```

### 9.2 고객의 비즈니스 안에서는 어떤 개념이 중요한가?

PM은 고객이 사용하는 업무 개념을 먼저 정리해야 한다.

이 단계에서는 고객의 말을 바로 시스템 용어, API, DB, 클래스명으로 바꾸지 않는다.  
고객이 이해하는 비즈니스 개념을 먼저 보존한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 고객 업무에서 반복적으로 등장하는 개념은 무엇인가?
- 고객이 관리하고 싶은 대상은 무엇인가?
- 고객이 중요하게 구분하는 상태는 무엇인가?
- 고객이 판단해야 하는 기준은 무엇인가?
- 고객 업무에서 중요한 예외 상황은 무엇인가?
- 고객이 사용하는 용어와 우리 시스템 용어가 다른 부분은 무엇인가?
```

이 단계의 산출물은 다음과 같다.

```text
- 고객 비즈니스 개념 목록
- 각 개념의 의미
- 개념 간 관계
- 고객이 중요하게 보는 상태값
- 고객 업무상 중요한 예외 상황
```

예시:

```text
고객 비즈니스 개념:
- 데이터 로딩 요청
- 로딩 작업
- 진행 상태
- 실패 상태
- 완료 상태
- 실패 원인
- 운영자 확인
- 재처리 필요 여부
```

### 9.3 이 시스템은 어떤 문제를 해결해야 하는가?

PM은 고객 요구를 기반으로, 이 시스템이 해결해야 하는 문제를 명확히 정의해야 한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 현재 고객 문제가 발생하는 이유는 무엇인가?
- 이 문제는 정보 부족 문제인가?
- 이 문제는 상태 관리 문제인가?
- 이 문제는 업무 흐름 문제인가?
- 이 문제는 권한/정책 문제인가?
- 사람이 수동으로 처리하던 부분은 무엇인가?
- 시스템이 자동화하거나 가시화해야 하는 부분은 무엇인가?
- 이 문제를 해결하지 않으면 어떤 업무 리스크가 생기는가?
```

이 단계의 산출물은 다음과 같다.

```text
- 문제 정의
- 문제 발생 원인
- 해결해야 하는 핵심 흐름
- 해결하지 않을 경우의 리스크
- 이번 스프린트에서 해결할 최소 문제 범위
```

예시:

```text
문제 정의:
현재 데이터 로딩 요청 이후 작업 상태가 시스템 안에서 명확히 드러나지 않는다.

문제 발생 원인:
로딩 작업이 독립적인 도메인 개념으로 관리되지 않고, 진행/실패/완료 상태가 사용자에게 노출되지 않는다.

해결해야 하는 핵심 문제:
로딩 작업을 시스템에서 추적 가능한 단위로 만들고, 운영자가 상태를 확인할 수 있게 해야 한다.
```

### 9.4 이 시스템에는 어떤 도메인 개념이 존재해야 하는가?

PM은 고객 비즈니스 개념을 시스템 안에서 관리 가능한 도메인 개념으로 정리해야 한다.

이 단계는 기술 설계가 아니다.  
이 단계는 시스템이 어떤 개념을 이해하고 관리해야 하는지 정의하는 단계다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 고객 비즈니스 개념 중 시스템 안에 독립적으로 존재해야 하는 것은 무엇인가?
- 어떤 개념은 상태를 가져야 하는가?
- 어떤 개념은 이력을 가져야 하는가?
- 어떤 개념은 정책이나 권한과 연결되는가?
- 기존 시스템 도메인으로 표현 가능한가?
- 새 도메인 개념이 필요한가?
- 기존 도메인 개념의 의미가 바뀌는가?
```

이 단계의 산출물은 다음과 같다.

```text
- 시스템 도메인 개념 목록
- 고객 비즈니스 개념과 시스템 도메인 개념의 매핑
- 새로 추가되는 도메인 개념
- 변경되는 기존 도메인 개념
- 상태값 후보
- 정책과 연결되는 도메인 개념
```

예시:

```text
고객 비즈니스 개념 → 시스템 도메인 개념

데이터 로딩 요청 → Loading Request
로딩 작업 → Loading Job
진행 상태 → Job Status
실패 원인 → Failure Reason
운영자 확인 → Operator Review
재처리 필요 여부 → Retry Requirement
```

### 9.5 이번 스프린트에서 어떤 기능을 다룰 것인가?

PM은 전체 요구사항 중 이번 스프린트에서 실제로 다룰 범위를 정해야 한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 이번 스프린트의 목표는 무엇인가?
- 고객 문제 중 이번에 반드시 해결해야 하는 것은 무엇인가?
- 이번 스프린트에서 사용자가 실제로 체감할 수 있는 변화는 무엇인가?
- 어떤 도메인 개념을 이번에 시스템에 반영할 것인가?
- 어떤 기능을 이번 스프린트에 포함할 것인가?
- 어떤 앱/서비스가 이번 스프린트에 참여해야 하는가?
- 각 앱/서비스는 어떤 역할을 맡아야 하는가?
- 이번 스프린트가 끝났을 때 무엇이 가능해져야 하는가?
```

이 단계의 산출물은 다음과 같다.

```text
- 스프린트 이름
- 스프린트 목표
- 포함 기능
- 참여 앱/서비스
- 앱/서비스별 역할
- 기대 결과
- 담당자 배정 후보
```

예시:

```text
Sprint:
데이터 로딩 상태 관리 1차

목표:
사용자가 데이터 로딩 작업을 생성하고, 운영자가 작업 상태를 확인할 수 있게 한다.

포함 기능:
- 로딩 작업 생성
- 로딩 상태 조회
- 실패/완료 상태 구분
- 운영자 확인용 상태 표시

참여 앱/서비스:
- Client
- API Gateway
- Loader Master
- Backend Service
```

### 9.6 이번 스프린트에서 제외할 것은 무엇인가?

PM은 이번 스프린트에서 하지 않을 것을 명확히 정해야 한다.

제외 범위가 명확하지 않으면 스프린트 범위가 계속 늘어나고, 담당자 배정 이후에도 혼란이 생긴다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 고객이 원하지만 이번 스프린트에서는 제외할 것은 무엇인가?
- 기술적으로 가능하지만 지금 넣으면 범위가 커지는 것은 무엇인가?
- 이후 스프린트로 넘겨야 할 기능은 무엇인가?
- 아직 정책 결정이 부족해서 제외해야 하는 것은 무엇인가?
- 도메인 개념은 필요하지만 구현은 미룰 수 있는 것은 무엇인가?
- 이번 스프린트의 성공과 직접 관련 없는 것은 무엇인가?
```

이 단계의 산출물은 다음과 같다.

```text
- 제외 기능 목록
- 제외 사유
- 후속 스프린트 후보
- 보류된 정책 결정
- 향후 확장 가능성
```

예시:

```text
이번 스프린트 제외 범위:
- 자동 재시도
- 로딩 성능 최적화
- 관리자 통계 대시보드
- 실패 원인 상세 분석
- Slack 알림 연동

제외 사유:
이번 스프린트의 핵심 목표는 로딩 작업을 시스템에서 추적 가능한 단위로 만들고, 상태를 조회할 수 있게 하는 것이다.

자동 재시도와 통계 대시보드는 상태 관리가 안정화된 이후 별도 스프린트에서 다룬다.
```

### 9.7 어떤 앱/서비스가 관련되는가?

PM은 기능을 개인별 세부 작업으로 바로 나누지 않는다.  
먼저 이번 스프린트에 관련되는 앱/서비스를 식별해야 한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 이 기능이 영향을 주는 앱/서비스는 무엇인가?
- 사용자와 직접 맞닿는 앱은 무엇인가?
- 요청을 중계하거나 조율하는 서비스는 무엇인가?
- 상태나 데이터를 관리하는 서비스는 무엇인가?
- 비동기 작업이나 백그라운드 처리가 필요한가?
- 외부 시스템 또는 알림 시스템과 연결되는가?
```

이 단계의 산출물은 다음과 같다.

```text
- 참여 앱/서비스 목록
- 각 앱/서비스가 이 스프린트에 참여해야 하는 이유
- 앱/서비스 간 관계
- 앱/서비스 간 의존성 후보
```

예시:

```text
참여 앱/서비스:
- Client
- API Gateway
- Loader Master
- Backend Service

참여 이유:
Client는 사용자 화면을 제공한다.
API Gateway는 Client 요청을 내부 서비스로 전달한다.
Loader Master는 로딩 작업 생성과 상태 관리를 담당한다.
Backend Service는 로딩 작업과 관련된 데이터 저장/조회/검증을 담당한다.
```

### 9.8 각 앱/서비스는 어떤 역할을 맡는가?

PM은 각 앱/서비스가 이번 스프린트에서 맡는 역할을 정리해야 한다.

이 역할 정의는 기술 구현 상세가 아니다.  
각 앱/서비스가 기능 완성에 어떤 책임을 가지는지 설명하는 수준이면 된다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 각 앱/서비스는 이번 스프린트에서 어떤 책임을 가지는가?
- 각 앱/서비스가 제공해야 하는 결과는 무엇인가?
- 각 앱/서비스가 다른 앱/서비스에 제공해야 하는 것은 무엇인가?
- 각 앱/서비스가 다른 앱/서비스로부터 기다리는 것은 무엇인가?
- 어떤 앱/서비스가 선행되어야 하는가?
- 어떤 앱/서비스가 후행으로 연결되는가?
```

이 단계의 산출물은 다음과 같다.

```text
- 앱/서비스별 역할 설명
- 앱/서비스별 책임 범위
- 앱/서비스별 기대 결과
- 앱/서비스 간 의존 관계
```

예시:

```text
Client:
사용자가 데이터 로딩을 요청하고, 로딩 상태를 확인할 수 있는 화면을 제공한다.

API Gateway:
Client 요청을 받아 Loader Master로 전달하고, 상태 조회 응답을 제공한다.

Loader Master:
로딩 작업 생성, 상태 관리, 실행 흐름 제어를 담당한다.

Backend Service:
로딩 작업과 관련된 데이터 저장, 조회, 검증을 담당한다.
```

---

### 9.9 각 앱/서비스 담당자는 누구인가?

PM은 앱/서비스 단위로 담당자를 배정한다.

PM은 세부 구현 Task를 직접 나누는 것이 아니라, 앱/서비스별 책임 범위에 담당자를 지정한다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 각 앱/서비스의 담당자는 누구인가?
- 한 사람이 여러 앱/서비스를 담당하는가?
- 담당자가 없는 앱/서비스가 있는가?
- 담당자 배정이 현재 팀 구조와 맞는가?
- 특정 담당자에게 일이 과도하게 몰리는가?
- 담당자에게 전달해야 할 요약 설명은 무엇인가?
```

이 단계의 산출물은 다음과 같다.

```text
- 앱/서비스별 담당자
- 담당자별 담당 영역
- 담당자별 간략 설명
- 담당자별 확인 링크
```

예시:

```text
Client:
담당자 A

API Gateway:
담당자 JaeCheon

Loader Master:
담당자 JaeCheon

Backend Service:
담당자 B
```

---

### 9.10 담당자에게 전달할 간략한 설명은 무엇인가?

PM은 저장 또는 확정 버튼을 누르기 전에, 각 담당자에게 전달될 설명을 확인해야 한다.

이 설명은 작업 지시서가 아니다.  
이 설명은 담당자가 “내가 이번 스프린트에서 어떤 영역을 맡았는지” 이해하기 위한 배정 요약이다.

PM이 확인해야 하는 것은 다음과 같다.

```text
- 담당자는 어떤 스프린트에 배정되었는가?
- 담당자는 어떤 앱/서비스를 맡는가?
- 담당자의 역할은 무엇인가?
- 담당자가 알아야 할 도메인 개념은 무엇인가?
- 이번 스프린트에서 기대되는 결과는 무엇인가?
- 함께 작업하는 앱/서비스는 무엇인가?
- 담당자가 링크로 들어왔을 때 어떤 내용을 보게 할 것인가?
```

이 단계의 산출물은 다음과 같다.

```text
- 담당자별 배정 요약
- 담당자별 Slack 알림 메시지
- 담당자별 확인 링크
```

예시:

```text
JaeCheon님, 다음 스프린트에 담당자로 배정되었습니다.

Sprint:
데이터 로딩 상태 관리 1차

담당 영역:
API Gateway / Loader Master

간략 설명:
Client 요청을 받아 Loader Master에 로딩 작업을 생성하고,
작업 상태를 조회할 수 있는 흐름을 담당합니다.

확인 링크:
https://app.example.com/assignments/123
```

## 10. PM이 설계하지 않는 것

Phase 1에서 PM은 기술 상세를 설계하지 않는다.

PM이 설계하지 않는 것은 다음과 같다.

```text
- API endpoint
- API request / response schema
- DB schema
- 테이블 구조
- 클래스 구조
- 함수 구조
- 프론트엔드 컴포넌트 구조
- 백엔드 내부 모듈 설계
- 구체적인 코드 구현 방식
- 테스트 코드 작성 방식
- 배포 파이프라인
- CI/CD 설정
```

이 항목들은 이후 개발자 실행 단계에서 다룬다.

## 11.5 AI 비판 / 검토 기능

PM은 작성 중인 내용을 AI에게 비판하거나 검토하게 할 수 있어야 한다.

요구사항:

```text
- 현재 고객 요구 요약이 실제 문제를 잘 반영하는지 비판해야 한다.
- 고객 비즈니스 개념 중 누락된 개념을 제안해야 한다.
- 시스템 도메인 개념이 고객 비즈니스 개념과 잘 매핑되는지 검토해야 한다.
- 문제 정의가 너무 넓거나 모호한 경우 지적해야 한다.
- 스프린트 포함 범위가 과도한 경우 경고해야 한다.
- 제외 범위로 빼야 할 후보를 제안해야 한다.
- 참여 앱/서비스 중 누락된 후보를 제안해야 한다.
- 앱/서비스별 역할이 모호한 경우 개선안을 제안해야 한다.
- 담당자에게 전달할 설명이 부족한 경우 보완안을 제안해야 한다.
```

AI 검토 결과는 다음 형식으로 제공한다.

```text
- 문제점
- 이유
- 수정 제안
- 반영 여부
```

## 11.12 CI를 통한 저장소 반영

PM이 저장 또는 확정 버튼을 누르면, 시스템은 PM 설계 내용을 Markdown 문서로 생성하고 CI를 통해 지정된 저장소에 반영해야 한다.

요구사항:

```text
- PM 설계 내용을 Markdown 문서로 생성해야 한다.
- 문서에는 고객 요구, 비즈니스 개념, 도메인 개념, 스프린트 범위, 앱/서비스 역할, 담당자 배정이 포함되어야 한다.
- 생성된 문서는 지정된 저장소 경로에 반영되어야 한다.
- 저장소 반영은 CI 파이프라인을 통해 수행되어야 한다.
- CI 실행 성공/실패 이력을 저장해야 한다.
- CI 실패 시 PM에게 실패 사유를 표시해야 한다.
```

주의사항:

```text
- 이 기능은 서비스 배포 목적의 CI/CD가 아니다.
- 이 기능은 PM 설계 문서를 저장소에 기록하기 위한 문서 반영 흐름이다.
```

## 17. Phase 1 이후 확장 방향

## 17.1 Phase 2: Worker / Developer Execution

Phase 2에서는 담당자가 자기 배정 내용을 확인하고 실제 작업을 수행한다.

포함 가능 기능:

```text
- 내 배정 목록
- 세부 작업 생성
- 작업 지시서
- 진행 상태 업데이트
- 질문 등록
- 블로커 등록
- 완료 보고
```

---

## 17.2 Phase 3: Work Orchestration & Synchronization

Phase 3에서는 PM 설계와 개발자 실행 상태를 연결한다.

포함 가능 기능:

```text
- 앱/서비스 간 의존성 추적
- 병목 감지
- 우선순위 재조정 추천
- PM 개입 필요 항목 표시
- 담당자별 다음 행동 추천
- 스프린트 전체 진행 상황 표시
```

## 18. 최종 정의

Phase 1은 PM이 요구사항을 기반으로 AI와 대화하며 고객 비즈니스 개념과 시스템 도메인 개념을 작성·비판·보완하고, 스프린트를 생성한 뒤, 앱/서비스 단위로 담당자를 배정하고, 저장/확정 시 Slack으로 각 담당자에게 배정 알림과 확인 링크를 전달하며, AI가 PM 설계 문서를 Markdown으로 생성해 CI를 통해 저장소에 반영하는 MVP다.

Phase 1은 작업 지시서 발행 시스템이 아니다.

작업 지시서는 Phase 2 또는 그 아래 개발자 실행 레벨에서 다룰 개념이다.
